# מחלקה: UserRepository

## 1. מידע כללי
*   **שם המחלקה:** `UserRepository`
*   **סוג:** מחלקה רגילה (Repository / Data Access Object)
*   **מטרת המחלקה:** מחלקה זו אחראית על כל פעולות בסיס הנתונים הקשורות למשתמשים. היא פועלת כמתווך בין ה-Activities לבין בסיס הנתונים SQLite, ומספקת מתודות נקיות כמו `registerUser` במקום כתיבת שאילתות SQL גולמיות בתוך ה-Activities.
*   **אינטראקציה:** בשימוש על ידי ה-Activity של ה-`LogAndReg` לבדיקת פרטי התחברות ושמירת משתמשים חדשים. היא משתמשת ב-`HelperDB` כדי לקבל גישה לבסיס הנתונים.

## 2. משתנים (שדות המחלקה)
| שם | סוג | מטרה | היכן בשימוש |
| :--- | :--- | :--- | :--- |
| `helperDB` | `HelperDB` | מנהל קובץ ה-SQLite. | בשימוש בכל מתודה לצורך פתיחת בסיס הנתונים. |

## 3. מתודות (שיטות) המחלקה

### שם המתודה: `findUser`
*   **סוג המתודה:** `public`
*   **ערך חזור:** `User` (אובייקט המשתמש שנמצא או `null`)
*   **פרמטרים:** `name (String)`, `password (String)`
*   **לוגיקה:**
    1. פותחת את בסיס הנתונים לקריאה.
    2. מריצה שאילתה למציאת שורה שבה השם והסיסמה תואמים.
    3. אם נמצאה שורה, היא משתמשת ב-`mapCursorToUser` כדי להפוך את השורה לאובייקט ג'אווה.
    4. סוגרת את החיבור ומחזירה את המשתמש.
*   **מתי נקראת:** כשהמשתמש מנסה להתחבר (Login).

### שם המתודה: `registerUser`
*   **סוג המתודה:** `public`
*   **ערך חזור:** `long` (ה-ID החדש של המשתמש בבסיס הנתונים)
*   **פרמטרים:** `user (User)`
*   **לוגיקה:**
    1. פותחת את בסיס הנתונים לכתיבה.
    2. אורזת את נתוני המשתמש בתוך אובייקט `ContentValues`.
    3. מורה לבסיס הנתונים להכניס את השורה החדשה.
    4. מחזירה את ה-ID שנוצר.
*   **מתי נקראת:** כשמשתמש חדש נרשם (Register).

### שם המתודה: `mapCursorToUser`
*   **סוג המתודה:** `private`
*   **לוגיקה:** קוראת נתונים מתוך `Cursor` (שהוא מעין מצביע לשורת תוצאה בבסיס הנתונים) ובונה ממנו אובייקט `User`.
*   **מה חשוב להבין:** זוהי מתודת עזר שנועדה לשמור על קוד נקי ולמנוע חזרתיות.

## 4. מחזור חיים (Lifecycle)
*   **לא רלוונטי:** קיימת כל עוד ה-Activity שמשתמשת בה קיימת.

## 5. אינטראקציה עם רכיבים אחרים
*   **בסיס נתונים SQLite:** היעד לכל אחסון הנתונים.
*   **מסך LogAndReg:** המשתמש המרכזי ברפוזיטורי הזה.

## 6. לוגיקה כללית של המחלקה
`UserRepository` הוא **פקיד תיוק**. כאשר Activity רוצה למצוא מישהו, היא מבקשת מהפקיד. הפקיד הולך למחסן (בסיס הנתונים), מחפש את התיקייה הנכונה, ומביא אותה חזרה כאובייקט `User`.

## 7. הסבר במילים פשוטות
חשבו על `UserRepository` כעל **מאבטח דיגיטלי**. ל-Activities אין את המפתח לכספת (בסיס הנתונים). הן שואלות את המאבטח: "האם יש משתמש בשם יוסי?". המאבטח בודק בכספת ומחזיר ל-Activity תשובה חיובית (אובייקט המשתמש) או שלילית (null).

---
**באגים/שיפורים:**
*   **אחסון סיסמאות:** כפי שצוין במחלקות אחרות, שמירת סיסמאות כטקסט פשוט היא סיכון אבטחתי. מתודת ה-`registerUser` צריכה באופן אידיאלי לבצע הצפנה (Hashing) לפני השמירה.
*   **בדיקת כפילות:** כרגע `registerUser` לא בודק אם שם המשתמש כבר תפוס. הבדיקה הזו מתבצעת ב-Activity, אך יהיה בטוח יותר לנהל זאת כאן או להגדיר אילוץ `UNIQUE` בבסיס הנתונים.
