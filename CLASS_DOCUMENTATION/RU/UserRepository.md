# Класс: UserRepository

## 1. Общая информация
*   **Имя класса:** `UserRepository`
*   **Тип:** Обычный класс (Репозиторий / DAO)
*   **Назначение:** Этот класс отвечает за все операции с базой данных, связанные с пользователями. Он выступает посредником между экранами приложения и базой данных SQLite, предоставляя понятные методы (например, `registerUser`) вместо написания сложных SQL-запросов в коде Activity.
*   **Взаимодействие:** Используется активностью `LogAndReg` для проверки учетных данных и сохранения новых пользователей. Использует `HelperDB` для доступа к файлу базы данных.

## 2. Переменные (поля класса)
| Имя | Тип | Цель | Где используется |
| :--- | :--- | :--- | :--- |
| `helperDB` | `HelperDB` | Управляющий файлом SQLite. | Используется в каждом методе для открытия базы данных. |

## 3. Методы класса

### Имя метода: `findUser`
*   **Тип метода:** `public`
*   **Возвращаемое значение:** `User` (объект найденного пользователя или `null`)
*   **Параметры:** `name (String)`, `password (String)`
*   **Логика работы:**
    1. Открывает базу данных для чтения.
    2. Выполняет запрос на поиск строки, где имя и пароль совпадают с введенными.
    3. Если строка найдена, использует метод `mapCursorToUser`, чтобы превратить эту строку в объект Java.
    4. Закрывает соединение и возвращает результат.
*   **Когда вызывается:** При попытке входа пользователя в систему.

### Имя метода: `registerUser`
*   **Тип метода:** `public`
*   **Возвращаемое значение:** `long` (новый ID пользователя в базе)
*   **Параметры:** `user (User)`
*   **Логика работы:**
    1. Открывает базу данных для записи.
    2. Упаковывает данные пользователя в объект `ContentValues`.
    3. Дает команду базе данных вставить новую строку.
    4. Возвращает сгенерированный базой ID.
*   **Когда вызывается:** При регистрации нового пользователя.

### Имя метода: `mapCursorToUser`
*   **Тип метода:** `private`
*   **Логика работы:** Считывает данные из «курсора» базы данных (это своего рода указатель на строку результата) и на их основе собирает объект `User`.
*   **Что важно понимать:** Это вспомогательный метод, который помогает держать код чистым и избегать повторений.

## 4. Жизненный цикл
*   **Неприменимо:** Существует до тех пор, пока существует экран, использующий этот объект.

## 5. Взаимодействие с другими компонентами
*   **База данных SQLite:** Конечная цель для хранения всех данных.
*   **Активность LogAndReg:** Основной потребитель данных этого репозитория.

## 6. Общая логика класса
`UserRepository` — это **«Заведующий архивом»**. Когда программе нужно кого-то найти, она обращается к заведующему. Тот идет в архив (базу данных), ищет нужную папку и приносит её в виде готового объекта `User`.

## 7. Упрощенное объяснение
Представьте `UserRepository` как **цифрового охранника**. У экранов приложения нет ключа от сейфа (базы данных). Они спрашивают охранника: «Есть ли у нас пользователь по имени Иван?». Охранник проверяет сейф и либо дает добро (объект пользователя), либо отрицательно качает головой (null).

---
**Советы по улучшению:**
*   **Хранение паролей:** Как уже отмечалось, хранение паролей в открытом виде небезопасно. Метод `registerUser` в идеале должен хешировать пароль перед сохранением.
*   **Проверка дубликатов:** Сейчас `registerUser` не проверяет, занято ли имя пользователя. Эта проверка делается в Activity, но было бы надежнее добавить ограничение `UNIQUE` в саму базу данных.
